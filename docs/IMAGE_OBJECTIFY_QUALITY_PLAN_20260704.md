## 0. 사용자 불만 원문 (재현 대상)

사진 1(원본)과 사진 2(객체화 결과) 대조:
1. 형태가 매끄럽지 않다 (픽셀 계단 + RDP 각짐)
2. 색이 원본과 다르다 (회색 채움이 검정/흰으로 뭉개짐)
3. 글씨가 매끄럽지 않다 (글자 형태 최대 보전 필요)
4. 선이 매끄럽지 않다

사진 3·4(원형 물체 이동 후): 공(M)이 "검정 원 폴리곤 + 흰 원 폴리곤 2겹 합성"으로 만들어져
일부만 옮기면 초승달 모양이 남음. "네이티브로 다루기 어려운 형태는 애초에 생성 안 되게 해야 하는가?"

사용자 제안(평가 대상): "모든 부분을 작은 점(소형 객체)으로 분해 생성" — 부분 수정 편의성 목적.

## 1. 근본 원인 (코드로 확인됨)

`js/image-vectorize.js` + `js/image-objectify.js`(커밋 ad629e3)가 원본의 회색조·원·직선·화살표·
점선을 전부 "테두리 없는 흑/백 폴리곤" 하나의 형태로만 찍어낸다. 그런데 렌더러(`js/render/`)는
이미 회색조 0~255 임의값(fillLevel), 테두리 있는 ellipse, 화살표/치수선/점선 달린 line, 부드러운
Catmull-Rom curve를 전부 네이티브로 지원한다 (`js/render/shapes.js`, `js/render/fill.js`,
`js/render/core.js`). **변환기가 렌더러가 이미 가진 표현력을 버리고 있다** — 이것이 불만 1·2·4·5의
공통 원인.

실측 확인(합성 이미지 테스트): 회색(밝기 200) 채움 영역이 Otsu 이진화 임계값에 따라 통째로
흰색 소실되거나 통째로 검정으로 뭉개짐 — `mask = gray <= threshold ? 1 : 0` 단일 임계값 구조상
3단계(흰·회·검) 그림을 원천적으로 표현 불가능.

## 2. 채택 확정된 개선안 (적대적 검증 통과 · needs-change 포함)

검증자가 "확정 성립(confirmed)" 또는 "요구되는 수정 포함(needs-change)"으로 판정한 안만 정리.
전부 `image-vectorize.js` / `image-objectify.js` 안에서 가능, 스키마·브랜치 규칙 비저촉 확인됨.

### 2-1. 회색조 보전 (불만 2) — **B4 안, 우선순위 1**

- 다단계 임계화(2단 Otsu 또는 히스토그램 k-means, K=2~4로 캡)로 흰/회색/검정 클래스 분리.
- 클래스 검증·병합 3종 필수(검증자 지적): (a) 잉크 대비 점유율 <1% 클래스 병합 (스캔 노이즈 방어),
  (b) 인접 클래스 대표값 차 ΔG<24 병합 (그라데이션 과분할 방어), (c) 안티앨리어싱 헤일로 판정 —
  해당 클래스 픽셀의 60% 이상이 더 어두운 클래스와 8-이웃 접촉 시 어두운 쪽에 흡수 (테두리 오검출 방어).
- **레벨별 컴포넌트 방출 게이트 필수(검증자 지적)**: 레벨 i에서 "그 레벨 고유 픽셀 수 < 컴포넌트 면적의
  ~2%"면 해당 레벨에서 방출 생략 — 안 그러면 순수 검정 도형도 모든 레벨에서 중복 방출됨.
- 누적 마스크(어두운 순 누적) → 기존 dilate→CC→traceContours 파이프라인 그대로 재실행,
  밝은 레벨 먼저 삽입(order 낮게), 어두운 레벨 나중(위) — 겹침으로 이음새 방지.
- fillLevel = 컴포넌트 내부 픽셀의 **원본 그레이 중앙값 실측치**로 교체 (현재 `image-objectify.js:348`
  `fillLevel: loop.isHole ? 255 : 0` 하드코딩 제거).
- **전제**: `binarize()`가 gray Uint8Array를 함께 반환하도록 `vectorizeImage` 반환 계약 확장 (B1·B4 공유).
- UI: "회색 단계 보존" 체크박스 1개(기본 on), off면 현행 단일 Otsu로 폴백.

### 2-2. 원/링 → 네이티브 ellipse (불만 1·5) — **B1 안, 우선순위 2**

- 판정은 RDP 이전의 **원시 경계점**(pre-dilate traceContours 출력)에서 수행 — dilate된 컴포넌트
  bbox 아님(검증자 지적: comp.bbox가 아니라 raw loop 자체 기준으로 명시할 것).
- 중심 = 경계점 무게중심, rx·ry = bbox 반폭/반높이. 정규화 잔차 `e_i = ((x-cx)/rx)²+((y-cy)/ry)²-1`,
  채택 기준 RMS(e)<0.08 AND max|e|<0.20.
- 축정렬 실패 시 PCA 회전 좌표계로 1회 재시도(회전 타원 → `ellipse.rotation` 필드 사용).
- 링(공 M) 판정: 바깥·구멍 루프 동심 조건(중심거리 <0.15·min(rx,ry)), 두께 t(θ) 36방향 샘플,
  std(t)/mean(t) < 0.3 → **ellipse 1개**: strokeWidth=평균두께(mm 환산), strokeLevel=링 잉크 그레이
  평균, fillLevel=구멍 내부 그레이 평균(대개 255).
- 구멍 없는 채움 원판: strokeWidth 0, fillLevel=잉크 그레이 평균.
- **효과**: "검정 도넛+흰 원 2폴리곤" 문제 원천 소멸 — ellipse 1객체라 일부만 옮겨지는 사고 불가능.

### 2-3. 코너 보존 스무딩 + 축 스냅 (불만 1·4) — **C1+C3 안, 우선순위 3, 검증자 "확정 성립"**

- `vectorizeImage`의 루프 처리(현재 273~279행 부근) 교체:
  1. collinearReduce 직후 원시 루프에서 각 꼭짓점의 지지방향각(전후 호길이 ~4px 구간 평균 진행방향 차)
     계산. θ>60°이며 지역 최대인 점 = 코너(비최대 억제 창 3px).
  2. 루프 시작점을 첫 코너로 회전 — 현재 rdp가 임의 시작점을 강제 keep해 곡선에 가짜 코너를
     만드는 부수 효과도 함께 제거.
  3. 코너-코너 스팬별 분류: 스팬 내 최대 수직거리 <0.75px → 직선(끝점 2개만), 아니면 곡선.
  4. **직선 스팬 후처리(C3)**: 각도 수평/수직 ±3° 이내면 축정렬 스냅(스팬 픽셀 평균으로 y/x 고정),
     인접 직선 스팬은 교점으로 공유 꼭짓점 재계산 — 사각형 모서리가 "진짜 4점"이 됨. 비축정렬 직선은
     최소자승 직선 피팅으로 끝점만 재투영.
  5. 곡선 스팬: 호길이 균등 재샘플(간격 3px) → **closed curve**로 삽입 (현행 closed polyline 대체).
     코너 앞뒤 1.5px 지점에 가드 앵커 2개 추가 — catmullRomClosedPath 제어점 오프셋이 (p2-p0)/6이므로
     코너 라운딩이 0.25px 이하로 억제됨(직각 뭉개짐 금지 제약 충족).
  6. 모든 스팬이 직선이면 현행 closed polyline 유지(사각형/삼각/블록/화살촉 — 직각 100% 보존).
     곡선 스팬 포함 시 closed curve로 삽입.
- 공(원)은 코너 0개인 순수 곡선 루프가 되어 매끈한 원으로 보임(2-2의 ellipse 피팅이 우선 적용되므로
  실제로는 피팅 실패한 불규칙 형태에만 이 경로가 쓰임).

### 2-4. 글자 보전 (불만 3) — **C4(정밀 벡터화) + C5(원본 크롭), 우선순위 4, C4는 검증자 "확정 성립"**

- **기본값 = C5 원본 크롭 이미지 배치**(정책 P3 결론): 글자 판정 영역을 원본에서 크롭한 dataURL로
  작은 `image` 객체로 배치(흰 배경→투명 처리). 무손실, 재라벨링 전제(§인수인계서)에 가장 적합.
  - **필요 수정(검증자 지적)**: 알파 규칙 — 회색조 경로는 RGB=(0,0,0) 고정 + alpha=255-gray
    (흰 배경 위 정확 재현). 유채색 감지 시에만 un-premultiply 분기.
  - 삽입 객체 필드는 `image-paste.js`의 insertImageObject와 동일 세트로 완비
    (mode/cutouts/aspectLocked/exportable 등).
- **옵션 = C4 정밀 벡터화**: `isText` 컴포넌트(`image-vectorize.js:282` 판정)에 한해 textEpsilon
  기본 0.3(도형용 슬라이더와 분리) + C1 코너 파이프라인(코너 임계 75°, 재샘플 1.5~2px) → closed curve.
  구멍 포함(ㅁ/o/4의 속). 도형 epsilon과 독립이라 도형 단순화를 올려도 글자는 안 망가짐.
- 기존 3모드(남기기/지우기/텍스트 대체)에 "원본 이미지로 유지(권장)" 옵션 추가.

### 2-5. 중심선·화살표·점선 → line (불만 4 완결) — **B2+B3 안, 우선순위 5 (가장 복잡, needs-change)**

- **B2 필요 수정(검증자 지적)**: 컴포넌트를 이분법(획 vs 영역)으로 나누지 말고, 컴포넌트 내부
  **거리변환(DT) 밴드 분해**로 교체 — 잉크 픽셀을 DT ≤ T_stroke(획 후보)와 DT > T_stroke(영역 코어)로
  나눠 각각 재-CC하고, 코어에서 지오데식 복원으로 가장자리 픽셀 회수 후 획 서브셋만 세선화 파이프라인에
  투입. 이래야 "공+지면선" "막대+치수선" 혼합 컴포넌트에서 선이 분리됨.
- junction(교차점)에서 획 분할 → 붙은 도형이 한 덩어리가 되는 문제 완화, 열린 line/polyline 객체로.
- **B3 필요 수정(검증자 지적)**: 화살촉 부착 시 반드시 `lineMode:"arrow"`로 전환(arrowVariant는
  arrowHead와 중복이라 미설정), 치수선은 `lineMode:"lengthArrow"` 사용(dimensionLabel 기본 "d" 렌더 확인됨).
- 일렬 짧은 collinear 조각 → dashLength/dashGap 있는 점선 line 1개로 병합(현재 minArea에 걸려
  사라지는 점선 보조선이 살아남게 됨).
- curve 타입은 arrowHead 미지원(스키마 문서 명시) — 화살표는 반드시 이 line 피팅 단계에서 흡수.

### 2-6. 통합 오케스트레이션 — **B5 안, 검증자 "확정 성립"**

- `image-vectorize.js`에 `fitPrimitives` 모듈 추가, `image-objectify.js`는 결과 타입별 삽입 분기만 확장.
- 사다리: [1] isText 게이트(글자는 피팅 대상 제외, 2-4 경로) → [2] 획/영역 분류(DT 기준) →
  [3] 영역 사다리: ellipse(2-2) → rect → triangle → closed polyline+meanGray 채움(2-1) →
  [4] 획 사다리: 세선화·junction 분할(2-5) → 화살촉 부착 → 대시 병합 → line/열린 polyline.
- 전역 계수 1개(`fitStrictness` 0.5~2.0)로 각 단의 채택 임계 스케일 — 기존 "곡선 단순화" 슬라이더처럼 UI 노출.
- **모든 단이 보수적 임계로 실패하면 정확히 현행 결과(+회색 채움 개선)로 수렴 → 회귀 위험 0에 가까움.**
  단별 독립 배포 가능(예: 2-1만 먼저 내보내는 것도 유효한 v0.X.0).
- 미리보기 오버레이를 판정 타입별 색으로 구분(타원=초록, 선=파랑, 사각=보라, 폴백 폴리곤=회색),
  클릭 시 "네이티브↔곡선↔이미지↔제외" 4단 순환으로 확장(정책 P3).

## 3. 정책 결론 (구현 시 반영)

### P1. "점 분해" 아이디어 — **반려 확정**

코드 근거: `js/render/scene.js:93` `scene.replaceChildren()`가 매 state.update마다 전체
재렌더 → 객체 수천 개면 드래그 중 프레임당 수 초. `js/transform.js` undo가 objects 배열 전체
JSON 딥클론 → 스냅샷당 수 MB. 그룹 클릭 시 selectedIds에 전 멤버 id, delete가 O(N×M).
점을 지워도 너덜너덜한 구멍만 남아 "부분만 깔끔히 삭제"라는 목적 자체 미달성.

**대안**: 컴포넌트(의미 단위)당 객체 1개 유지 + 삽입물을 거대 단일 그룹 대신 **컴포넌트별 소그룹**으로
분리, 기존 더블클릭 멤버 지목(`tools.js:309-315`, e.detail>=2 → targetedId, 주황 테두리) UX를
그대로 활용. 긴 폴리라인은 코너/junction에서 분할해 선분 단위 삭제 granularity 확보(2-5의 부산물).

### P2/P3. "다루기 어려운 형태는 생성 금지" — **폴백 사다리로 대체**

"생성 안 함" 대신 사다리: ① 피팅 성공 → 네이티브 타입, ② 실패했지만 단순 → curve/polyline,
③ 복잡·저신뢰 및 글자 → **원본 크롭 image 삽입**(§2-4). "건너뛰고 경고"·"매번 사용자 선택"은
기본값 아님 — 시험 그림에서 무단 누락이 최악의 실패 모드, 결정 피로도 있음.
