## ★ 시작 전에 — "어디부터 이어서 하면 되는가" (반드시 먼저 읽을 것)

이 작업은 **여러 세션에 걸쳐 이어서** 진행됩니다. 앞선 세션(Claude 또는 이전 Codex 실행)이
일부를 이미 끝냈을 수 있습니다. **아무 코드도 쓰기 전에, 지금 어디까지 되어 있는지부터
확인하십시오.** 순서는 이렇습니다:

1. **바로 아래 "작업 진행 현황(체크포인트)" 표를 읽으십시오.** 각 단계(E/D/F)의 상태가
   `☐ 미착수` / `◐ 진행중` / `☑ 완료`로 적혀 있습니다. 이 표가 진행 상황의 1차 기준입니다.
2. **표를 그대로 믿지 말고 git 로그로 교차 확인하십시오**: `git log --oneline -20`을 실행해
   `feat: ...(E)...`, `feat: ...(D)...` 같은 커밋이 실제로 있는지 보십시오. 표가 `☑ 완료`인데
   해당 커밋이 없다면, 또는 반대라면 git 로그를 신뢰하십시오(표 갱신을 빠뜨렸을 수 있음).
3. **`◐ 진행중`인 단계가 있으면 거기서부터 이어서** 하십시오. 그 단계의 "여기까지 됨 / 남은 것"
   메모를 표 아래에서 읽고, 어떤 파일이 반쯤 작성됐는지 `git status`와 `git diff`로 확인한 뒤
   이어가십시오. **완료된 단계(☑)를 다시 손대지 마십시오.**
4. 모든 단계가 `☐ 미착수`면 처음부터(E → D → F) 시작하십시오.
5. **한 단계를 끝낼 때마다, 커밋하기 직전에 아래 체크포인트 표를 갱신하십시오**(상태 아이콘 +
   "여기까지 됨" 메모 + 관련 커밋 해시). 이 표를 최신으로 유지하는 것이 다음 세션을 돕는
   핵심입니다. 표 갱신도 그 단계 커밋에 함께 포함시키십시오.

## 1. 프로젝트 개요

- 이름: **5E** (sciEnceEducationalExamEasyEditor) — 과학 교사가 시험문제용 그림(물리·화학·생명·지구)을
  그리는 순수 프런트엔드 웹앱. 빌드 도구 없음, ES 모듈을 브라우저가 직접 로드.
- 실행: `5E_hub` 폴더에서 `run-server.bat` 더블클릭(또는 `python -m http.server 8100`) →
  `http://localhost:8100/`.
- **당신에게는 브라우저가 없습니다.** 문법 검증은 `node --check js/파일.js`로, 순수 로직은
  `node tests/*.mjs`로 확인하십시오(Node 내장 assert만 씀, 프레임워크 없음). UI 동작은 코드를
  꼼꼼히 리뷰하는 것으로 대체하고, 각 단계 마지막에 **사람이 브라우저로 확인해야 할 체크리스트**를
  남기십시오.
- 파일 구조 핵심:
  - `index.html` — 전체 DOM 골격(버튼·모달 마크업 대부분 여기 하드코딩)
  - `js/state.js` — 전역 상태 스키마(단일 스토어), `js/main.js` — 모듈 배선 진입점
  - `js/object-types.js` — **오브젝트 타입 레지스트리**(한 줄 추가로 여러 모듈에 동시 반영되는 설계)
  - `js/tools.js` — 도구 선택·드래그 생성 파이프라인(`SHAPE_TYPE` 맵이 핵심)
  - `js/render/scene.js`의 `renderObject()` — 타입별 렌더 스위치
  - `js/inspector/section-*.js` — 우측 인스펙터 섹션들(타입별 파일), `js/inspector.js`가 조립
  - `js/personal-objects.js`, `js/ui-dialogs.js` — 최근 추가된 참고용 신규 기능(패턴 참고하십시오)
  - `css/style.css`, `css/inspector.css`, `css/exam-library.css`
  - `tests/*.mjs` — Node 단위 테스트(순수 함수 대상)

### 1단계(E) — 기본값 설정에 "인쇄 비교 이미지" 등록 기능

**배경**: `js/exam-preview.js`는 이미 "그림을 실제 시험지 위 실제 인쇄 크기로 미리 보는" 기능이
있습니다(파일 상단 주석 참고). 다만 배경 이미지가 `assets/exam-backgrounds/sample-physics-8jeol.png`
하나로 하드코딩돼 있습니다. 사용자가 원하는 것은: **`설정 ▾ → 기본값 설정` 모달에서 본인이 가진
이미지(예: 실제로 인쇄해 본 시험지 스캔본)를 직접 등록**하고, 인쇄 미리보기(`exam-preview.js`)를
열 때 그 등록된 이미지들 중에서 배경을 골라 비교할 수 있게 하는 것입니다.

**요구사항**:
1. `js/settings.js`의 `기본값 설정` 모달(`buildModal()`)에 새 섹션 "인쇄 비교 이미지"를 추가:
   - 파일 선택 버튼(PNG/JPG) → 등록. 이름(선택, 기본값 파일명), **실제 인쇄 크기(mm, 가로·세로)**를
     함께 입력받으십시오(스캔 이미지의 픽셀 크기와 실제 종이 크기가 다르므로, 이게 있어야
     `exam-preview.js`의 "실제 크기로 얹기" 로직과 맞습니다 — 기존 8절 배경의 물리 크기를
     `exam-preview.js`에서 참고해 기본값으로 제안하십시오).
   - 등록된 이미지 목록: 작은 썸네일 + 이름 + 삭제 버튼(위 2번 요건대로 축소된 삭제 버튼 +
     `showConfirm` 확인창 패턴을 `personal-objects.js`에서 그대로 가져오십시오).
   - 저장은 `localStorage` 새 키(예: `5e.previewBackgrounds`, JSON 배열:
     `[{id, name, dataUrl, widthMm, heightMm}]`)로. dataURL이 커질 수 있으니 업로드 시 캔버스로
     리사이즈해(예: 긴 변 2000px 이하) 용량을 억제하십시오.
   - `js/settings.js`의 `PERSONAL_KEYS`(또는 동급 목록)에 새 키를 추가해 "설정 저장하기" 위젯의

### 2단계(D) — 공통 도구에 "자" · "각도기" 가이드 추가

**요구사항**:
1. 새 오브젝트 타입 하나(`type: "guide"`, `kind: "ruler" | "protractor"`)를 `object-types.js`에
   등록(§2-5 패턴). `sizeBox: 1`을 켜서 기존 이동/리사이즈/락 인프라를 재사용하십시오.
2. **공통 도구 팔레트**(`index.html`의 "공통 도구" 그리드, `js/tools.js`)에 자·각도기 버튼 2개를
   추가하고, 기존 `RECT`/`O`처럼 **드래그로 생성**되게 하십시오(`SHAPE_TYPE` 맵에 등록,
   `makeShape()`에 분기 추가).
   - 자(ruler): 드래그한 사각형의 가로 길이를 자의 길이로 씁니다.
   - 각도기(protractor): 드래그한 사각형의 대각선(또는 가로)을 반지름으로 하는 반원으로 생성.
3. **눈금 간격은 드래그 크기와 무관하게 고정값**을 씁니다(요구사항 원문: "드래그 해서 생성한다고
   해서 눈금 간격은 변해서는 안되고"). 즉 렌더러는 `obj.tickIntervalMm`(자) /
   `obj.tickIntervalDeg`(각도기) 값을 읽어 그 간격으로 눈금을 그리고, 드래그는 오직 길이/반지름만
   결정합니다. 새로 생성될 때의 초깃값은 3단계 아래 "기본값 설정"에서 가져옵니다.
4. 렌더러(`js/render/scene.js`의 `renderObject()`에 `case "guide"` 추가, 실제 그리기는
   `js/render/shapes.js`류에 함수로 분리해도 됩니다):
   - 자: 긴 직선(또는 얇은 띠) + `tickIntervalMm`마다 짧은 눈금선, 10칸마다(또는 적당한 간격) 숫자
     라벨.
   - 각도기: 반원 호 + `tickIntervalDeg`마다 방사형 눈금선, 30°마다 각도 숫자 라벨.
   - 둘 다 `obj.opacity`(0~1)를 지원(기존 `image` 타입의 opacity 처리를 그대로 참고, 0~1 float를
     SVG `opacity` 속성에 매핑).
5. **인스펙터 섹션 신설**(`js/inspector/section-guide.js`, `section-image.js` 템플릿 참고):
   - 눈금 간격 입력(자=mm, 각도기=도)
   - 투명도 슬라이더(0~100%, `section-image.js`의 opacity 슬라이더 패턴 그대로)
   - **보호 섹션(위치 고정/오브젝트 잠금)은 이미 타입 무관 공통 로직**(`section-protect.js`,
     `o.locked`/`o.positionLocked`)이므로 guide 타입도 자동으로 동작해야 합니다 — 별도 구현 불필요,
     동작 확인만 하십시오.
6. **기본값 설정**(`js/settings.js` 기본값 모달)에 "자 눈금 간격(mm)", "각도기 눈금 간격(°)" 입력
   필드 2개를 추가하고, `DEFAULTS_KEY` 페이로드에 포함시켜 새 guide 오브젝트 생성 시 이 값을
   초깃값으로 쓰게 하십시오.
7. 스냅(`js/snap.js`)·자르기(`cut-tool.js`) 등 기존 범용 로직은 타입을 몰라도 되면(예: `SHAPE_TYPES`
   플래그 기반으로 동작) 손대지 마십시오. 만약 `guide` 타입이 이런 로직에서 예외를 던진다면
   최소한의 가드만 추가하고, 그 외 동작 확장은 하지 마십시오(범위 최소화).

**완료 기준**:
- `node --check` 전체 통과, 기존 `tests/*.mjs` 회귀 없음.
- 코드 리뷰로 드래그 생성 → 렌더 → 인스펙터 편집 → 위치고정/잠금 → 삭제/undo 흐름 확인.
- 사람 확인 체크리스트 남기기(자 눈금이 실제로 mm 단위로 정확한지는 브라우저에서 눈금자 도구와
  나란히 놓고 확인 필요).

### 3단계(F) — 문항 세트 관리 (엑셀 탭 방식 멀티 아트보드)

**배경**: 지금 이 앱은 프로젝트 하나 = 캔버스 하나(오브젝트 목록 하나)입니다. 교사가 한 회차 시험
20문항의 그림을 그릴 때, 문항마다 새 프로젝트 파일을 여닫아야 하는 불편이 있습니다. 요구사항은
"엑셀 시트 탭"처럼 **한 프로젝트 파일 안에 여러 개의 독립된 캔버스(탭)를 두고, 하단/상단 탭 바로
전환**하는 것입니다.

**설계 지침 (중요 — 반드시 이 방식으로 구현하십시오, 임의로 아키텍처를 바꾸지 마십시오)**:

앱 전체가 `state.objects` / `state.layers` / `state.undoStack` / `state.redoStack` /
`state.viewBox`를 최상위에서 직접 읽는 구조입니다(수십 개 모듈이 `state.get().objects`를 호출).
이 참조 방식을 탭 인식(tab-aware)으로 바꾸려면 모든 모듈을 건드려야 해서 위험이 너무 큽니다.

**대신 이렇게 하십시오**: 탭 전환 시 최상위 필드(objects/layers/undoStack/redoStack/viewBox 등
"문서 내용"에 해당하는 필드)의 **내용물을 통째로 교체**하는 방식을 쓰십시오. 즉:

```
state.tabs = [
  { id, name, objects, layers, activeLayerId, undoStack, redoStack, viewBox },
  ...
]
state.activeTabId = "..."
```

탭을 전환할 때:
1. 현재 `state.objects/layers/activeLayerId/undoStack/redoStack/viewBox`를
   `state.tabs`에서 `activeTabId`에 해당하는 항목에 다시 써 넣는다(퇴장하는 탭 저장).
2. 전환해 들어갈 탭의 저장된 값을 `state.objects/layers/...`에 그대로 복사해 넣는다.
3. `state.activeTabId`를 바꾸고 `state.update()`로 구독자에게 알린다(기존 render/applyViewBox
   구독이 자동으로 다시 그림).

이렇게 하면 **다른 모듈은 전혀 수정할 필요가 없습니다** — 그들은 여전히 `state.get().objects`만
읽고 있고, 그 내용물이 탭 전환 시점에 바뀔 뿐입니다. `selectedIds`는 탭 전환 시 빈 배열로
초기화하십시오(다른 탭의 선택 상태가 새 탭에 남아있으면 혼란).

**요구사항**:
1. `js/state.js`: 초기 상태에 `tabs: [{id:"tab1", name:"1번", objects:[], layers:[...], ...}]`,
   `activeTabId: "tab1"` 추가. 기존 최상위 objects/layers 등은 그대로 두되, 이 초깃값이
   `tabs[0]`의 내용과 동기화되게 하십시오(첫 로드 시 하위호환).
2. **탭 바 UI**: 캔버스 하단(또는 상단, 격자 컨트롤 바 근처)에 엑셀 시트 탭처럼 가로로 나열된 탭
   목록 + `+`(새 탭 추가) 버튼. 각 탭은 더블클릭 또는 우클릭 메뉴로 **이름 변경**, 우클릭 메뉴 또는
   버튼으로 **복제**·**삭제**(마지막 탭 하나 남았을 때 삭제 방지 + `showConfirm`). 클릭으로 전환.
   새 파일 `js/document-tabs.js`를 만들어 이 로직을 담고, `main.js`에서 배선하십시오
   (`initToolHint`, `initPersonalObjects` 배선 방식을 참고).
3. **Undo/Redo는 탭별로 독립**이어야 합니다(위 설계대로 `undoStack/redoStack`이 탭마다 따로
   저장되므로 자동으로 만족됩니다 — `js/transform.js`의 undo/redo 함수는 여전히
   `state.get().undoStack`만 보고 동작하면 되고, 그 배열의 identity가 탭마다 다를 뿐입니다).
4. **프로젝트 저장/불러오기**(`js/project-io.js`): 지금은 단일 문서를 직렬화할 것입니다 —
   이제 `state.tabs` 전체(+ `activeTabId`)를 저장/복원하도록 확장하십시오. **하위호환**: 옛 형식
   (탭 필드 없는 파일)을 불러오면 그 내용을 `tabs: [{id:"tab1", name:"1번", ...옛 내용}]` 하나짜리
   배열로 감싸 로드하십시오(기존 저장 파일이 깨지면 안 됩니다).
5. **일괄 내보내기**(파일 메뉴에 "전체 탭 내보내기" 추가): 모든 탭을 순회하며 각각 PNG로
   내보냅니다. **UI를 매번 전환하며 깜빡이게 하지 말고**, `js/svg-export.js`의
   `rasterizeExportCanvas(s, opts)`가 받는 `s` 인자에 현재 `state.get()` 대신 **각 탭의 스냅샷
   객체**(`{...state.get(), objects: tab.objects, layers: tab.layers}` 형태)를 직접 만들어
   넘기는 방식으로 구현하십시오(함수 시그니처를 보면 이미 `s`를 파라미터로 받게 되어 있어
   이 방식이 자연스럽습니다). 파일명은 탭 이름을 파일명-안전 문자로 치환해 사용하거나,
   `순번_탭이름.png` 규칙을 쓰십시오. 여러 파일을 한 번에 내려받는 것이 브라우저 제약으로 번거로우면,
   zip으로 묶는 라이브러리를 새로 들여오지 말고(외부 의존성 추가 금지) **순차 다운로드**(각 파일마다
   `downloadBlob` 호출, 브라우저가 여러 다운로드를 허용하는지는 별개 이슈이니 100ms 정도 간격을 두고
   순차 트리거)로 구현하십시오.
6. 탭 이름 규칙: 사용자가 직접 정하게 두되, 새 탭 기본 이름은 `번호+1번`(예: 기존 탭이 3개면
   "4번")으로 자동 제안하십시오.

**완료 기준**:
- `node --check` 전체 통과.
- **새 Node 테스트를 `tests/`에 추가**하여 최소한 다음을 순수 함수 단위로 검증하십시오(브라우저
  없이도 검증 가능한 부분): 탭 전환 시 내용 스왑 로직, 옛 형식(탭 없는) 프로젝트 파일을 불러왔을 때
  1개 탭으로 감싸지는 로직. `js/document-tabs.js`에서 이 로직을 **DOM과 분리된 순수 함수**로 뽑아
  테스트 가능하게 설계하십시오(예: `swapTabContent(state, fromId, toId)`,
  `wrapLegacyProjectAsTabs(oldProjectJson)`).
- 코드 리뷰로: 탭 추가/이름변경/복제/삭제/전환, 저장→불러오기 왕복(옛 형식 포함), 일괄 내보내기
  흐름을 확인.
- 사람 확인 체크리스트를 남기십시오(특히: 탭 전환이 실제 브라우저에서 순간적으로 이전 탭 내용이
  깜빡이며 보이는지, undo가 탭을 넘나들며 오작동하지 않는지는 실제 조작 확인이 꼭 필요합니다).
