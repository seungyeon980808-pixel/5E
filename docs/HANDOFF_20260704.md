## 4. ★ 오늘(07-04) 내린 중요 결정: 이미지→객체 "API 폐기"

**결정: 외부 비전 API + 서버리스 프록시 파이프라인을 계획에서 뺀다.** 이유·근거·후속 조치:

- **왜 빼나**: image-dev에서 **PNG→편집가능 객체 변환을 로컬로 직접** 만들고 있으므로 API가 할 일이 없어짐.
  API를 빼면 비용·키 노출·배포/유지·**시험지 이미지 외부전송 개인정보 문제**가 전부 사라짐. "무료·누구나" 취지와 일치.
- **되돌리기 어려운 결정이 아님 (핵심)**: 원래 설계가 이걸 대비해 뒀다. `image-to-object-v1` JSON 스키마 +
  `js/image-import-mock.js`(검증·좌표변환·미리보기·삽입·한번의Undo)는 **"JSON을 누가 만드느냐"와 "어떻게 삽입하느냐"를
  분리한 이음새(seam)**다. API는 그 JSON을 만드는 여러 생산자 중 하나였을 뿐. → 지금 빼도 잃는 것 0,
  훗날 로컬 정확도가 부족하면 API를 "또 다른 생산자"로 붙이면 됨(재작업 0).
- **image-dev 세션에 전할 것(가장 중요)**:
  1. PNG 변환기의 **출력을 `image-to-object-v1` JSON 형태로** 내보내라.
  2. 삽입은 **기존 `js/image-import-mock.js` 경로를 재사용**하라(검증·미리보기·Undo가 이미 다 있음).
     파일명이 "mock"이라 버릴 코드처럼 보이지만 실제로는 **삽입 백엔드**다. "파일에서 읽기"를
     "메모리 JSON 받기"로 살짝 일반화하면 그대로 쓴다(이참에 이름 mock→import 권장).
  3. 로컬 벡터화 원형 `js/image-objectify.js`가 이미 있음(임계값·Hough — 곡선/텍스트 약함). 개선/신규 여부는
     image-dev 판단이되 **출력은 반드시 v1 JSON으로 모을 것**(안 그러면 삽입 경로가 3개로 쪼개짐).
- **참고 문서**: `docs/IMAGE_TO_OBJECT_API_DESIGN_20260630.md`(스키마·매핑규칙·UX 설계 — API 부분은 이제 보류),
  `docs/qa-fixtures/image_to_object_mock_v1.json`(스키마 예시).

## 9. AI 작업 배분 가이드 (사용자 제공 원문 + 현재 프로젝트 적용)

> 사용자가 별도로 관리하던 `AI_작업배분_가이드.md`를 이 인수인계에 통합. 새 세션은 이 절만 봐도 모델 선택 가능.

### 9.0 원칙
**"틀리면 비싼 것 = 비싼 모델, 틀려도 고치기 쉬운 것 = 저렴한 모델."**
난이도가 아니라 **되돌리는 비용**이 기준. (예: 태그 오타=저렴, 데이터 스키마 실수=비쌈.)

모델 선택보다 아래 둘이 토큰 절약에 더 크게 기여 — 항상 먼저 챙긴다:
1. **세션을 작업 단위로 끊는다.** 긴 대화는 누적 맥락 때문에 매 응답 비용이 커진다. 큰 기능 끝나면
   결과를 요약 문서(이 문서 같은)로 남기고 새 세션을 연다.
2. **작업 지시를 명확하게.** 모호한 지시로 여러 번 왕복하는 게 비싼 모델보다 큰 낭비. 인수인계·정의를 그대로 붙여 시작.

### 9.1 모델 계층 (2026-07 현재 이름 — 작업 전 `/model`로 재확인)
`/model`로 현재 선택 가능 목록을 확인하고, **이름이 아니라 "계층상 위치"로 매핑**해서 쓴다.

| 계층 | 현재 이름(대략) | 이 프로젝트 용도 |
|---|---|---|
| 가장 저렴/빠름 | **Haiku 4.5** (`claude-haiku-4-5`) | 단순 반복, 포맷 변환, `?v=` 일괄 상향, 태깅 초안 |
| 중간(구현 주력) | **Sonnet 5** (`claude-sonnet-5`) | 명세 정해진 기능 구현, 코드 리뷰, 테스트, 일반 코딩 |
| 고성능 | **Opus 4.8** (`claude-opus-4-8`) | 어려운 디버깅, 복잡한 리팩토링, 여러 파일 구조 변경 |
| 최상위 | **Fable 5** (`claude-fable-5`) | 전체 아키텍처 설계, 방향 결정, "되돌리기 비싼" 판단 |

### 9.2 이 프로젝트 작업별 배분
- **Fable/Opus급 — 설계·판단(세션 시작 1회, 이후 넘기기)**: 스키마를 건드릴지 판단, 인덱스/태그 구조 설계,
  객체화 모듈을 스토어 어디에 연결할지, 여러 모듈 리팩토링 계획. **완료되면 결정을 이 문서에 기록하고 구현은 아래로 넘긴다.**
- **Sonnet급 — 구현(실제 작업 대부분)**: 알고리즘 이식, 객체화 옵션 UI, 검색 UI·lazy load, 일괄 태깅 UI,
  정해진 스키마의 변환 코드, 원인 명확한 버그.
- **Opus급 — 어려운 디버깅(Sonnet이 막힐 때만 승격)**: 여러 모듈 걸친 원인불명 버그, Undo 히스토리 꼬임,
  좌표변환(px↔mm) 특정 케이스 오차. **규칙: 항상 Sonnet 먼저, 막히면 시도내역 요약해 Opus 승격.**
- **Haiku급 — 단순 반복**: 2000개 그림 태깅 보조(파일명 파싱→초안 태그), 포맷팅·주석·`?v=` 상향, 테스트 뼈대, 문서 정리.

### 9.3 적용 방법
- 세션 전체: `claude --model opus` / `--model sonnet`. 세션 중 전환: `/model`(단, 히스토리 재로드=캐시무효라
  **작업 단계가 바뀌는 지점에서만** 전환).
- **서브에이전트별 모델 고정(추천)**: `.claude/agents/*.md` frontmatter에 `model: haiku|sonnet|opus` 지정해두면
  반복 작업에 자동으로 저렴한 모델이 붙음.
- 프로젝트 기본값: `.claude/settings.json`의 `model`에 Sonnet급을 박고, 설계·디버깅 때만 잠깐 올렸다 내림.

### 9.4 요청 전 체크리스트
1. 되돌리기 비싼가? → 상위 모델 + 이 문서 먼저 갱신.
2. 명세가 이미 문서에 있나? → 그대로 인용(재설명 금지).
3. 세션이 너무 길어졌나? → 큰 기능 끝나면 요약 후 새 세션.
4. 반복 작업이면 서브에이전트로 만들 가치 있나? → 있으면 한 번 만들어 재사용.

### 9.5 현재 상황에 대한 구체 지침
- **리팩토링(Day 2~3)·API폐기 판단(Day 4 앞부분)은 끝났다.** 남은 Day 4 나머지·Day 5·프론트엔드는
  **명세가 정해진 구현/반복** → **Sonnet급이 기본**, 막히는 디버깅에만 Opus 승격. Fable은 새 설계 결정이 생길 때만.
- 최상위(Fable) 접근이 제한되면 Opus로 대체하고 설계 단계에 시간을 더 들인다.
